## Solution statement

The gaming industry is a massive global industry, reaching users from young children to adults. Games are no longer only for entertainment; they are also used for learning, social connection, team-building, and online communities. However, developing a complete multiplayer game still takes significant time, technical effort, and coordination. Even a simple card game becomes difficult when it needs real-time interaction, hidden player information, rule enforcement, and cross-device access. This creates a clear challenge: how can developers turn a game idea into a working multiplayer experience faster, while still keeping the game fair, secure, and easy to use?
Our project addresses this challenge by building a real-time multiplayer version of Monopoly Deal. Traditional card games are enjoyable because they combine strategy, surprise, and social interaction. However, when they are moved into a digital format, several problems appear. Players should not be able to see opponent cards or deck order. Game rules must be enforced correctly. Multi-step actions, such as rent payment, stealing, property changes, and “Just Say No” responses, must be handled without confusion. At the same time, users should be able to join easily without installing complex software.
To solve this, we developed a browser-based Monopoly Deal game using an authoritative server architecture. The server acts as the single source of truth for the game. It manages the deck, validates every move, controls game flow, and sends each player only the information they are allowed to see. This protects hidden information and prevents unfair play. Players can join simply by opening a browser and entering the host’s local IP address, making the game accessible across different devices and operating systems.
The user experience is designed to be simple and natural. Players drag cards from their hand into the correct play area, such as the bank, property section, or action area. The system automatically understands the card type and routes it correctly. For more complex actions, guided pop-up windows help the player select targets, choose payment cards, move wildcard properties, or respond to another player’s action. All game updates happen in real time, so every player sees the latest valid game state immediately.
The main value of our solution is that it shows how a complex multiplayer game can be developed quickly while maintaining fairness, usability, and strong rule enforcement. Within 24 hours, we were able to bring the project to a significant working level, including real-time multiplayer gameplay, browser-based access, server-side rule validation, hidden-information protection, and guided user interactions. Setup time is reduced to under two minutes, illegal moves are automatically blocked, and players do not need to manually check rules during the game. The project also creates future opportunities for remote play, AI opponents, spectator mode, replay systems, and tournament support.
Overall, this solution fits the judging criteria because it is feasible as a working prototype, innovative in its secure multiplayer game design, well-designed through a simple browser-based user flow, and effective in showing that a significant multiplayer gaming prototype can be achieved within 24 hours using IBM Bob.

## Tech statement

We used IBM Bob as the main technical development partner for our real-time multiplayer Monopoly Deal application. Bob supported the project from requirement analysis to working prototype by helping us design and implement the main software architecture, shared type system, real-time server, multiplayer synchronization layer, frontend interaction flow, testing logic, and debugging process. The development was organized into 8 Bob-assisted phases, covering shared data contracts, server initialization, game-loop implementation, networking, action-card mechanics, frontend UI, and system validation.
We used multiple IBM Bob MCP modes/tools during development. Ask mode was used to reason about the Monopoly Deal rule flow, hidden player information, turn sequencing, and valid action paths. Plan mode helped divide the system into client, server, and shared modules, and define the implementation order. Code mode was used to generate and refine TypeScript, React, Node.js, Express, and Socket.IO code. Advanced mode was used for the most complex technical parts, including authoritative server logic, state sanitization, recursive “Just Say No” interrupt chains, payment resolution, card action routing, and real-time event handling.
Bob’s output was integrated into the shared TypeScript layer, where it helped define enums, card categories, action types, event contracts, constants, and game-state models. These shared contracts allow both the frontend and backend to use the same structure for player state, card objects, actions, payments, socket events, and game phases. This reduced desynchronization risk between client and server.
On the backend, Bob helped implement the Node.js and TypeScript authoritative game server. This included the GameEngine, TurnManager, CardFactory, WinCondition, CSV deck-loading pipeline, action-handler system, payment handler, reaction handler, Socket.IO event router, and state sanitizer. The server validates every move, controls turn progression, hides opponent hands and deck order, and broadcasts only sanitized game-state updates to each connected player.
On the frontend, Bob helped implement the React and TypeScript browser client. This included the game board, player hand, card renderer, opponent view, bank and property areas, center piles, turn indicator, drag-and-drop card flow, and interaction modals for target selection, rent payment, reactions, and dual-use card decisions.
Bob was also used for technical testing and debugging. It helped create test cases, validate edge cases, and identify a duplicate-discard bug. The final system reached a 96% test pass rate, with 24 out of 25 tests passing. We did not integrate watsonx.ai or watsonx Orchestrate into the application logic. This version used IBM Bob and Bob MCP modes as the primary IBM technologies to reach a significant working multiplayer prototype within 24 hours.
